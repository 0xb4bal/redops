## Summary

[2-3 sentences: what is the bug, where, what can an attacker do]

Example: An Insecure Direct Object Reference (IDOR) vulnerability in the profile
endpoint allows an authenticated attacker to access or modify any user's private
data by manipulating the `user_id` parameter.

---

## Impact

[Concrete business impact. What data? Which users? What operations?]

An attacker with a valid account can:
1. Read private profile data of any user (name, email, address, payment info)
2. Modify any user's account settings
3. This affects all N registered users

---

## Steps to Reproduce

**Setup:**
- Account A (attacker): attacker@email.com
- Account B (victim): victim@email.com (use your own test account)

**Steps:**
1. Log in as Account A
2. Navigate to `https://api.example.com/v1/users`
3. Intercept the request in Burp Suite
4. Modify `user_id=YOUR_ID` to `user_id=VICTIM_ID`
5. Forward the request
6. Observe: response contains Account B's private data

## Root Cause

The server does not validate that the authenticated user is authorized to
access the requested user ID. Authorization check is missing in:
`[endpoint/function if known]`

---

## Remediation

1. Implement server-side authorization: verify the authenticated user owns the
   requested resource before returning data
2. Use indirect object references (map user-visible IDs to internal ones)
3. Add automated tests for IDOR in CI/CD pipeline
